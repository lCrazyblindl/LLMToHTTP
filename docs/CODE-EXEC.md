# Real Anthropic code-execution head-to-head (v0.4 · R6)

_By [`experiments/code_exec_real.py`](../experiments/code_exec_real.py), 2026-07-01, model `claude-haiku-4-5-20251001`._

**What this is.** Anthropic's real server-side **code-execution** tool (`code_execution_20250825` - the version Claude Haiku 4.5 supports; no beta header for the tool itself) against our own local-sandbox `code_exec` variant and the naive full-list baseline, on the exact task Stage 15's live matrix already validated (`T2_count_females`: "How many of all the animals are female?").

**The real constraint that shapes this comparison.** Anthropic's code-execution container has **no internet access** - it cannot call a live API itself. Neither can our own local sandbox, which is hard-wired to an in-process pet-zoo client rather than a real network call. So this isn't "who can reach a live API" (neither can, from inside the sandbox) - it's "given the same data, does real code-execution collapse bucket C the same way ours does, without the raw data ever re-entering the model's visible context?" Both mechanisms get the data the same way in spirit: ours via an injected `zoo` object in a local subprocess; Anthropic's via a **Files API upload + `container_upload`** - the documented pattern for handing the sandbox data without it passing through the model's token stream.

- tokenizer / usage: **real, billed** `usage.input_tokens`/`usage.output_tokens` for the real code-execution row; naive and ours are cited from `experiments/token-bench/validation.md (Stage 15 live matrix, k=3 mean, Haiku)` rather than re-spending tokens to re-measure already-real numbers.

## Result

| form | mechanism | total tokens | correct | source |
| --- | --- | ---: | --- | --- |
| `openapi_full` (naive) | full 50-record list re-enters context | 6121 | (validated) | Stage 15 live matrix |
| `code_exec` (ours) | local subprocess sandbox, injected `zoo` client | 1636 | (validated) | Stage 15 live matrix |
| **code-execution (real Anthropic)** | server-side sandbox, data via `container_upload` | **15613** | yes — "I'll read the animals.json file and count how many records have gender 'female'. Now I'll count how many records have gender 'female': Let me use the correct path: 24" | this run, live, billed |

**Read — a genuinely humbling, honest result.** Real code-execution's total (15613 tokens: 15202 in + 411 out) came in **above both** the naive full-list baseline (6121) **and** our own local sandbox (1636) — the opposite of what the `code_exec` thesis predicts, on this one run. The content-block trace shows why: `['server_tool_use', 'text_editor_code_execution_tool_result', 'server_tool_use', 'bash_code_execution_tool_result', 'server_tool_use', 'bash_code_execution_tool_result']` — the model **viewed the uploaded file first** (a `text_editor_code_execution` read), got the wrong path on its first bash attempt, and only counted correctly on a second try (its own final text says "Let me use the correct path"). Viewing the file returns its content as a tool result that re-enters context exactly like a naive fetch would — plus the retry added a whole extra round.

This surfaces a real, structural difference between the two "compress bucket C" mechanisms this project has now measured on the real API: **Tool Search's saving is structural** — `defer_loading` is enforced by the server regardless of what the model does (R5 confirmed this: ~90% saved automatically). **Code-execution's saving is behavioral** — it only materializes if the model's *own* generated code computes the answer without printing the raw data anywhere along the way; our local sandbox enforces that discipline structurally (only the `result` variable ever leaves the subprocess, the model never *can* print the raw list back), which is precisely why ours stayed small (1636) while the real tool, free to `view`/retry, did not (15613) — on this one run. Caveat: k=1 for the real row (one live call, one model, one task) — noisy, and a model that goes straight to a counting script without viewing the file first would likely land far closer to (or under) our own number; this isn't a claim that real code-execution is inherently worse, only that its saving isn't guaranteed the way `defer_loading`'s is. Also: neither sandbox can call a *live* external API directly (no internet in Anthropic's container; ours is hard-wired to an in-process client) - a live-API-backed comparison is out of scope here and would need its own security review before extending our own sandbox that way.
